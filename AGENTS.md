# AGENTS.md

默认使用 strict opencode 模式。

## 输出规则
- 只输出纯文本或纯代码块。
- 不要输出 patch、diff、Edit 摘要、花式终端格式。
- 所有命令必须可直接复制粘贴。
- 所有 shell 命令必须使用标准 ASCII 字符。
- 长内容可分段输出，但不要省略关键内容。
- 不要只给结论，优先给证据、最终结果和验证方法。

## 文件操作规则
- 修改文件前先读取文件内容。
- 小改动优先精确修改。
- 必要时才整文件重写。
- 每次修改后，必须再次读取并输出完整最终文件内容。
- 不要只说“已修改”或“已更新”。
- 如涉及密钥、token、apiKey、密码，必须提醒用户轮换或遮盖。

## 路径与目录规则
- 不要预设固定项目路径。
- 不要写死配置文件、脚本、日志、工作区或服务目录，除非已经确认。
- 如果路径不明确，优先按以下顺序处理：
  1. 搜索文件名
  2. 浏览当前工作区
  3. 询问用户输入完整路径
  4. 使用最近操作过的文件
- 适配不同机器、不同目录结构、不同项目名称。

## 提问与菜单规则
- 当任务描述过宽时，优先使用编号菜单，而不是开放式追问。
- 菜单尽量通用，不要写死固定路径。
- 对危险操作必须先确认，例如：
  - 删除文件
  - 覆盖关键配置
  - kill 进程
  - 重启服务
  - 替换密钥、令牌或密码
- 对配置修改，先确认要改哪个字段，再执行。
- 如果用户已经说得足够明确，不要重复追问。

## Shell / 运维执行规则
- 优先给最小可执行命令。
- 复杂任务分步骤执行，并明确每一步目的。
- 不要把多个高风险动作盲目合并成一大段命令。
- 不要默认吞掉错误，除非用户明确要求。
- 如果命令执行失败，要解释失败点，而不是直接假设成功。
- 优先提供可验证结果的命令，例如：
  - 查看日志
  - 查看进程
  - 查看端口
  - 查看配置
  - 查看最终文件内容

## 通用修复流程
处理配置、服务、脚本、代理、机器人、接口或其他系统异常时，默认按以下顺序执行，不要跳步。

### 第 1 阶段：诊断
- 先读取当前现状，不要立刻修改。
- 优先使用项目自带或系统自带的诊断命令。
- 先给出证据，再下结论。
- 不要只根据“状态正常”判断已经修好。

### 第 2 阶段：补缺
- 检查缺失项，例如：
  - 缺失目录
  - 缺失文件
  - 缺失配置
  - 缺失注册项
  - 缺失环境变量
  - 缺失依赖
- 发现缺失项时，先说明缺什么，再补。
- 补缺后要展示补完结果。

### 第 3 阶段：补全
- 检查已有配置是否不完整，例如：
  - 字段缺失
  - 规则缺失
  - 权限不完整
  - fallback 缺失
  - 地址、密钥、变量未配置
- 补全后必须输出完整最终文件内容。
- 不要只输出变更摘要。

### 第 4 阶段：增强
- 在基础功能恢复后，再做增强，例如：
  - 安全加固
  - fallback 配置
  - 自动备份
  - 更清晰的菜单
  - 更好的日志检查命令
- 不要在故障未修复前优先做增强。
- 增强优先服务于稳定性、可维护性和安全性。

### 第 5 阶段：验证
- 每次关键修改后必须验证。
- 验证优先包括：
  - 服务是否运行
  - 配置是否生效
  - 日志是否正常
  - 目标功能是否实际恢复
- 不要只说“应该可以了”。
- 必须尽量提供实际证据。
